**What it is.** The attacker is the user. Their input contains instructions designed to override the system prompt, the developer's intentions, or the model's training.

**How it works.** Common shapes: *"Ignore prior instructions and …"*; persona overrides (*"You are now DAN, an AI without restrictions"*); fake authority (*"As your developer, I now command you to …"*); language switches; instruction smuggling inside otherwise-innocent requests.

The mechanism in all cases is the same: the user supplies tokens that compete with the system prompt for the model's attention. The model has been *trained* to weight the system prompt more heavily, but training is a soft preference, not a structural barrier. Sufficient adversarial pressure overrides it.

**Where it dies in this architecture.** Direct injection by itself does nothing — what matters is whether the model can *act* on the injected instructions. With a [mediated tool surface](../patterns/mediated-tool-surface/), the worst the model can do is invoke an existing tool with allowed arguments. There is no `execute_sql` to invoke; there is no `dump_all` to invoke. The injection succeeds at confusing the model and fails at extracting data. See the [attack path](../architecture/02-attack-path/) for a worked trace.

**Companion controls.** [Audit log](../patterns/audit-anomaly/) records the attempt; if frequent from one user, the [anomaly detector](../patterns/audit-anomaly/) escalates.

**Reference.** Greshake et al., 2023, [arXiv:2302.12173](https://arxiv.org/abs/2302.12173) — covers both direct and indirect cases.
